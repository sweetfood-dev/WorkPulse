# PRODUCT_SENSE.md

## Purpose

이 문서는 모호한 요구사항에서 제품 판단을 어떻게 내릴지 정의합니다.

## Core Rules

- 내부 구조보다 사용자 목표를 먼저 본다
- 기본 경로를 단순하게 유지한다
- 옵션을 늘리기 전에 사용자가 실제로 얻는 가치를 본다
- 지금 하지 않아도 되는 것은 범위 밖으로 남긴다

## Good Product Judgment

좋은 판단은 아래를 만족합니다.

- 누구의 어떤 문제를 푸는지 명확하다
- 성공 기준이 설명 가능하다
- 범위가 통제된다
- 실패 비용을 줄인다

## Required Evidence

- 대상 사용자 또는 행위 주체가 이름 붙여져 있다
- 개선하려는 핵심 사용자 목표나 문제를 한 문장으로 설명할 수 있다
- 성공을 어떻게 확인할지 최소 한 가지 관찰 가능한 결과가 있다
- 의도적으로 하지 않는 범위가 적혀 있다

## Blocking Questions

- 이 변경으로 누가 무엇을 더 쉽게 하게 되는가
- 기본 경로가 더 단순해지는가, 아니면 복잡도가 늘어나는가
- 범위 밖으로 남긴 것은 무엇인가
- 기존 정신 모델을 깨면 전환 비용이 설명돼 있는가

위 질문에 답하지 못하면 구현을 바로 시작하지 말고 spec 또는 plan으로 승격합니다.

## Escalate When

- 사용자군의 목표가 충돌할 때
- 성공 기준이 모호할 때
- 구현 비용은 큰데 가치가 불분명할 때
- 기존 정신 모델을 깨는 변화가 있을 때
