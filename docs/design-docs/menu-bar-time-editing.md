# menu-bar-time-editing.md

## 문제와 맥락

WorkPulse 메인 popover는 `Current Session`을 중심으로 오늘 상태를 빠르게 읽게 하는 dashboard여야 한다.
동시에 사용자는 오늘 출근/퇴근 시간을 같은 popover 안에서 수정할 수 있어야 한다.

여기서 핵심 선택지는 두 가지였다.

- 시간을 항상 바로 편집 가능하게 둘지
- 기본은 읽기 전용으로 두고, 필요할 때만 수정 상태로 전환할지

## 목표

- 메뉴바 popover의 glanceability를 유지한다
- 오늘 기록 수정 흐름을 한 화면 안에서 끝낸다
- 실수 수정, 의도치 않은 값 변경, transient popover 특유의 닫힘 리스크를 줄인다

## 비목표

- 전체 설정 창 설계
- 과거 기록 수정 UX
- 모든 입력을 자동 저장하는 범용 패턴 정의

## 대안 비교

### 대안 A. 항상 inline editable

장점:
- 바로 수정 가능하다
- 모드 전환이 없다

단점:
- 기본 dashboard 스캔성이 떨어진다
- AppKit 시간 입력 컨트롤이 좁은 popover에서 항상 노출되면 시각적으로 무겁다
- 바깥 클릭으로 popover가 닫힐 때 편집 손실이나 혼란이 생길 수 있다
- 라이브로 갱신되는 `Current Session`과 편집 컨트롤이 같은 위계에서 경쟁한다

### 대안 B. 별도 설정 창 또는 별도 화면에서 수정

장점:
- popover를 아주 단순하게 유지할 수 있다
- 입력 제어를 크게 만들기 쉽다

단점:
- `한 화면 안에서 끝나는 수정`이라는 요구를 만족하지 못한다
- 오늘 기록의 짧은 수정에 비해 흐름이 과해진다

### 대안 C. 같은 popover 안의 읽기 전용 기본 상태 + edit mode

장점:
- 기본 상태를 dashboard처럼 깔끔하게 유지한다
- 필요한 경우에만 입력 컨트롤을 노출한다
- popover라는 제약을 유지하면서도 별도 창 없이 수정이 가능하다
- 저장 시점과 취소 시점을 명확하게 둘 수 있다

단점:
- 한 번의 추가 액션이 필요하다
- popover 내부 상태 전환 설계가 필요하다

## 선택한 방향

`대안 C. 같은 popover 안의 읽기 전용 기본 상태 + edit mode`를 선택한다.

구체 기본값:

- 기본 상태는 읽기 전용이다
- `Start Time`, `End Time` 영역에서 수정 진입이 가능하다
- edit mode에서는 선택한 row가 같은 자리에서 입력 상태로 전환된다
- 별도 하단 편집 패널을 확장하지 않는다
- time input UI와 명시적 적용/취소 액션을 보여준다
- 저장 후 popover는 조회 상태로 돌아오고, 현재 세션/집계를 다시 계산한다
- 편집 중에도 `Current Session`과 요약 영역은 화면에 남겨 두어 상태판 성격을 유지한다

## 선택 이유

- 이 popover의 1순위 목적은 `빠른 상태 확인`이다
- 수정은 중요하지만 보조 행동이다
- transient 메뉴바 popover에서는 auto-save보다 명시적 적용이 더 안전하다
- Stitch 메인 화면의 정보 위계를 유지하기 쉽다
- 같은 자리 전환은 compact하고, popover 폭을 늘리거나 정보 구조를 바꾸지 않아도 된다

## 후속 리스크

- edit mode의 진입 affordance가 약하면 사용자가 수정 가능성을 놓칠 수 있다
- 적용/취소 액션이 불분명하면 저장 시점이 헷갈릴 수 있다
- 추후 과거 날짜 수정까지 요구가 커지면 이 패턴만으로는 부족할 수 있다

그 경우에는 별도 기록 편집 화면 또는 설정 창으로 확장할지 다시 판단한다.
